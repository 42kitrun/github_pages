# 생성형 AI 서비스 개인정보처리방침, 무엇을 더 써야 하나

_written by Codex, Claude_

오늘 개인정보처리방침을 다시 쓰다가 개인정보보호위원회의 2026년 4월 개정판을 찾았다. 원래 보려던 것은 일반적인 처리방침 작성 기준이었는데, 목차에서 **생성형 AI 서비스 처리방침 부록**이 먼저 눈에 들어왔다.

API 하나 붙여 챗봇을 만드는 일은 이제 어렵지 않다. 문제는 그 순간 사용자가 입력한 프롬프트, 첨부파일, 생성 결과물, 모델 학습 여부, 해외 API 사업자와의 위탁 관계가 한꺼번에 개인정보처리방침의 대상이 된다는 점이다. 이번 개정은 그 지점을 별도 부록으로 꺼내 정리했다.

![2026년 4월 생성형 AI 서비스 개인정보 처리방침 개정 흐름|697](/assets/posts/security/ai-service-privacy-policy-2026/2026_AI_privacy_policy_infographic.svg)

## 2026년 4월 개정에서 달라진 점

개정 내용 전체를 보면 생성형 AI만 다룬 것은 아니다. 수탁자나 제3자 제공 대상이 많고 자주 바뀌는 경우 유형별 기재를 허용하는 기준, 처리방침 변경사항 안내 방식, 온디바이스 처리 기준, 수탁자의 처리방침 작성 기준, 행태정보와 간이형 처리방침 기준도 함께 정리됐다.

그중 기술 블로그에서 따로 볼 만한 부분은 생성형 AI 서비스 부록이다. 일반 웹서비스의 처리방침은 보통 `수집 항목`, `처리 목적`, `보유 기간`, `제3자 제공`, `처리 위탁`을 채우면 골격이 잡힌다. 생성형 AI 서비스는 여기에 다음 질문이 추가된다.

- 사용자가 입력한 프롬프트와 첨부파일을 저장하는가
- AI가 만든 답변이나 이미지 같은 결과물을 저장하는가
- 입력값이나 결과물을 모델 학습에 사용하는가
- 사용자가 학습 활용을 거부할 수 있는가
- 외부 LLM API로 데이터가 넘어가는가
- 부적절한 답변에 대해 신고하거나 이의를 제기할 수 있는가

즉, 생성형 AI 서비스의 처리방침은 "무엇을 수집하는가"에서 끝나지 않는다. 입력된 데이터가 모델과 운영 시스템 안에서 어떤 경로로 쓰이는지를 설명해야 한다.

![생성형 AI 서비스 개인정보처리방침 핵심 정리](/assets/posts/security/ai-service-privacy-policy-2026/AI_service_privacy.png)

## 적용 대상은 자체 모델 개발사만이 아니다

부록을 읽을 때 가장 먼저 지워야 할 오해가 있다. 이 기준은 거대한 AI 모델을 직접 학습시키는 회사만을 위한 것이 아니다.

다음과 같은 서비스도 모두 검토 대상이다.

|구분|예시|처리방침에서 봐야 할 지점|
|---|---|---|
|외부 LLM API 연동|OpenAI, Anthropic, Google API로 챗봇 제공|처리 위탁, 국외 이전, 입력 데이터 보관 여부|
|공개 모델 추가 학습|오픈 모델을 서비스 데이터로 파인튜닝|학습 목적, 학습 데이터 범위, 거부권|
|자체 AI 기능 운영|문서 요약, 고객 상담, 이미지 생성 기능|입력값·결과물 수집, 보유 기간, 안전조치|
|플러그인·외부 도구 연동|예약, 결제, 검색, 업무 자동화 도구 호출|제3자 제공 또는 위탁 여부 구분|

작은 서비스라도 사용자의 자연어 입력을 받아 AI API로 보내면 개인정보 처리 흐름이 생긴다. 사용자가 이름, 연락처, 건강 정보, 계약 내용, 고객 상담 내용을 프롬프트에 넣을 수 있기 때문이다.

## 처리방침에 추가해야 할 핵심 항목

이미지 두 장의 내용을 합치면 핵심은 여섯 가지로 정리된다.

|항목|처리방침에 적어야 할 내용|
|---|---|
|의도된 용례|AI 기능이 누구를 대상으로 어떤 목적과 범위에서 동작하는지|
|입력 데이터|텍스트, 음성, 이미지, 첨부파일 등 프롬프트 입력값의 수집·저장 여부|
|생성 결과물|답변, 요약문, 이미지, 음성 등 AI 산출물의 저장 여부|
|학습 활용|입력값이나 결과물을 모델 학습, 파인튜닝, 품질 개선에 쓰는지|
|선택권과 거부권|학습 활용 동의, 철회, Opt-out 경로와 처리 결과|
|안전성과 책임|민감정보 입력 주의, 필터링, 접근통제, 신고·이의제기 절차|

일반적인 문구인 "서비스 제공 및 품질 개선"만으로는 부족하다. 예를 들어 고객 상담 챗봇이라면 "이용자 문의에 대한 자동 응답 제공", "상담 품질 개선을 위한 대화 로그 분석", "동의한 이용자 입력 데이터의 모델 성능 개선 활용"처럼 목적을 나눠 써야 한다.

## 프롬프트는 별도 수집 항목으로 보는 편이 안전하다

생성형 AI 서비스에서 가장 놓치기 쉬운 항목은 프롬프트다. 프롬프트는 단순한 검색어가 아니다. 이용자가 직접 입력한 자유 형식 데이터이고, 그 안에는 개인정보나 민감정보가 섞일 수 있다.

처리방침에는 최소한 다음을 구분해 적어야 한다.

|데이터|예시|확인할 내용|
|---|---|---|
|직접 입력 정보|프롬프트 텍스트, 음성, 이미지, PDF, 문서 파일|수집 여부, 저장 여부, 처리 목적|
|생성 결과물|AI 답변, 요약문, 생성 이미지, 변환 파일|저장 여부, 재사용 여부|
|자동 생성 정보|IP, 쿠키, 기기 정보, 이용 기록, 오류 로그|일반 서비스 로그와 AI 기능 로그의 관계|

그리고 입력창 주변에도 별도 안내가 필요하다. 주민등록번호, 여권번호, 카드번호, 의료정보처럼 입력하지 말아야 할 정보를 처리방침 안에만 숨겨두면 실제 사용자가 보기 어렵다. 처리방침에는 원칙을 쓰고, 제품 화면에는 짧은 주의 문구를 두는 편이 낫다.

## 학습 활용 여부는 가장 명확하게 써야 한다

AI 서비스 처리방침에서 가장 민감한 문장은 "입력 데이터를 학습에 사용하는가"다.

사용하지 않는다면 그렇게 쓰면 된다. 외부 API 사업자의 엔터프라이즈 계약이나 API 정책상 입력 데이터가 모델 학습에 쓰이지 않는 구조라면, 그 사실은 이용자에게 중요한 정보다.

사용한다면 범위를 나눠야 한다.

- 어떤 데이터를 학습에 쓰는가
- 원문 그대로 쓰는가, 가명처리나 필터링 후 쓰는가
- 어떤 목적의 학습인가
- 얼마 동안 보관하는가
- 이용자가 동의하지 않거나 철회하면 어떻게 되는가
- 이미 학습에 반영된 데이터의 삭제가 제한될 수 있는가

특히 Opt-out은 메뉴 이름까지 적는 것이 좋다. "고객센터로 문의"보다 "설정 > 개인정보 > AI 학습 활용 거부"처럼 실제 접근 경로가 드러나야 사용자가 권리를 행사할 수 있다.

## 외부 API를 쓰면 위탁과 국외 이전을 같이 봐야 한다

서비스 내부에 `chat completions` 호출 하나가 추가됐을 뿐이어도 법적 구조는 단순하지 않다. 사용자의 입력이 외부 AI 사업자에게 전달된다면 보통 개인정보 처리업무 위탁 여부를 검토해야 한다. 해외 사업자의 서버에서 처리된다면 국외 이전 항목도 함께 검토해야 한다.

처리방침에는 다음 정보가 들어가야 한다.

|구분|기재 예시|
|---|---|
|수탁자|AI API 제공 사업자명|
|위탁 업무|생성형 AI 기반 질의응답, 요약, 분류, 이미지 생성 등|
|이전 국가|데이터가 처리·보관되는 국가|
|이전 항목|프롬프트, 첨부파일, 생성 결과, 이용 기록 등|
|이용 목적|AI 기능 제공, 장애 분석, 보안 모니터링 등|
|보유 기간|API 사업자의 보관 기간 또는 즉시 삭제 여부|

여기서 처리위탁과 제3자 제공도 구분해야 한다. 자사 서비스 기능을 제공하기 위해 LLM API를 호출하는 것은 위탁에 가깝다. 반면 AI가 외부 예약 서비스, 결제 서비스, 제휴 상담 서비스로 사용자의 정보를 넘겨 그 회사가 독립적으로 처리한다면 제3자 제공에 가까워질 수 있다.

## 제품 구현과 처리방침은 같이 맞아야 한다

처리방침은 문서 작업처럼 보이지만 실제로는 제품 명세에 가깝다. 처리방침에 Opt-out을 제공한다고 쓰려면 설정 화면이나 고객센터 프로세스가 있어야 한다. 학습에 쓰지 않는다고 쓰려면 로그 저장, 벡터 DB 적재, 분석 파이프라인, 외부 API 설정까지 그 설명과 맞아야 한다.

개발자가 먼저 확인할 체크리스트는 다음과 같다.

```text
[ ] AI 기능의 목적과 대상 사용자를 한 문장으로 설명할 수 있는가
[ ] 프롬프트, 첨부파일, 생성 결과물의 저장 위치를 알고 있는가
[ ] 입력 데이터가 로그, 분석 도구, 벡터 DB, 모델 학습 파이프라인으로 흘러가는지 확인했는가
[ ] 외부 LLM API 사업자의 데이터 보관·학습 정책을 계약 기준으로 확인했는가
[ ] 학습 활용 동의와 거부, 철회 경로가 실제 제품에 있는가
[ ] 민감정보·고유식별정보 입력 방지 안내와 필터링 정책이 있는가
[ ] 부적절한 답변에 대한 신고 또는 이의제기 창구가 있는가
[ ] 처리방침 변경 시 이용자에게 알릴 기준과 방식이 정해져 있는가
```

## 정리

이번 개정에서 눈에 띄는 점은 생성형 AI를 특별한 기술로만 보지 않고, 개인정보 처리 흐름이 복잡한 서비스 유형으로 다뤘다는 것이다. 핵심은 어렵지 않다. 사용자가 무엇을 입력하고, 그 데이터가 어디로 가며, 저장되는지, 학습에 쓰이는지, 거부할 수 있는지를 공개하라는 내용이다.

개발자 입장에서는 처리방침을 마지막 배포 단계에서 쓰면 늦다. AI 기능을 설계할 때부터 로그 보관 기간, 외부 API 설정, 학습 데이터 분리, Opt-out 경로를 같이 정해야 한다. 그래야 처리방침이 실제 시스템과 어긋나지 않는다.
