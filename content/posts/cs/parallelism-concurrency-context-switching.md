# 병렬성과 동시성, 그리고 컨텍스트 스위칭 — 단일 코어 CPU가 여러 일을 처리하는 법

_written by Claude-Code_

## 배경: 사람도, CPU도 사실은 싱글코어다

사람은 원래 단일 처리 구조다. 지금 이 글을 읽으면서 동시에 음악을 듣고, 커피를 마시고, 옆 사람과 대화를 나누는 것처럼 느껴지지만, 뇌의 고차원 인지 처리는 실제로 한 번에 하나씩 집중하며 빠르게 전환할 뿐이다. 멀티태스킹처럼 *보이는* 것이 실은 빠른 *전환*이다.

초기 컴퓨터의 CPU도 마찬가지였다. 단일 코어로 여러 프로그램을 동시에 실행해야 한다는 과제 앞에서, 운영체제는 영리한 해법을 고안했다. 실제로 동시에 실행하는 게 불가능하다면, 동시에 실행하는 *것처럼* 보이게 만들면 된다. 여기서 **동시성(Concurrency)** 이라는 개념이 탄생한다.

이 글에서는 병렬성과 동시성이 어떻게 다른지, 그리고 동시성을 구현하는 두 핵심 메커니즘인 컨텍스트 스위칭과 시분할이 어떻게 작동하는지를 명확히 정리한다.

## 1. 병렬성(Parallelism) — 실제로 동시에 실행한다

병렬성은 **물리적으로 동시에** 여러 작업을 실행하는 것이다. 멀티코어 CPU 환경에서만 가능하다. 코어 A가 작업 A를 처리하는 동안 코어 B는 작업 B를, 코어 C는 작업 C를 독립적으로 실행한다.

```
코어 1: [작업 A ─────────────────]
코어 2: [작업 B ─────────────────]
코어 3: [작업 C ─────────────────]
         ↑ 실제로 동시에 실행 중
```

**목적**: 처리량(Throughput) 극대화. 더 많은 일을 더 빠른 시간 안에 끝낸다.

**비유**: 세 명의 요리사가 각자의 주방에서 동시에 서로 다른 요리를 만든다. 요리사 수만큼 요리 속도가 선형적으로 증가한다.

병렬성은 강력하지만 코어가 많을수록 비용이 따른다. 멀티코어 CPU가 필요하고, 작업을 독립적으로 분리할 수 있어야 한다. 공유 자원이 있다면 동기화 문제(Race Condition, Deadlock)가 발생할 수 있다.

---

## 2. 동시성(Concurrency) — 동시에 처리되는 것처럼 설계한다

동시성은 **논리적으로** 여러 작업이 동시에 진행되는 것처럼 보이는 구조다. 싱글코어에서도 가능하다. 실제로는 한 번에 하나의 작업만 실행하지만, 매우 빠르게 번갈아가며 처리함으로써 사용자에게는 동시 실행처럼 느껴진다.

```
코어 1: [작업 A]─[작업 B]─[작업 C]─[작업 A]─[작업 B]─...
         ↑ 실제로는 하나씩 순서대로, 그러나 빠르게 교대
```

**목적**: 반응성(Responsiveness) 향상. 어떤 작업이 I/O 대기 중이어도 다른 작업이 CPU를 사용할 수 있다.

**비유**: 한 명의 요리사가 국을 끓이는 동안 기다리는 대신, 그 사이에 샐러드를 준비하고 고기를 굽는다. 동시에 하는 게 아니라, 대기 시간을 낭비하지 않도록 교대로 처리한다.

### 병렬성 vs 동시성 핵심 비교

| 구분 | 병렬성(Parallelism) | 동시성(Concurrency) |
|------|-------------------|-------------------|
| 실행 방식 | 물리적 동시 실행 | 논리적 교대 실행 |
| 필요 조건 | 멀티코어 CPU | 싱글코어도 가능 |
| 목적 | 처리량 향상 | 반응성 향상 |
| 관점 | 하드웨어 수준 | 소프트웨어 설계 수준 |

> 멀티코어 환경에서는 병렬성과 동시성이 함께 작동한다. 각 코어가 병렬로 실행하면서, 각 코어 위에서도 동시성이 적용된다.

## 3. 컨텍스트 스위칭(Context Switching) — 동시성을 구현하는 메커니즘

동시성은 저절로 실현되지 않는다. CPU가 작업 A에서 작업 B로 전환할 때, 나중에 작업 A를 정확히 이어서 실행할 수 있도록 현재 상태를 어딘가에 저장해야 한다. 이 과정이 **컨텍스트 스위칭**이다.

### 컨텍스트 스위칭 과정

```
현재: 프로세스 A 실행 중
  ↓
① CPU 레지스터, PC(Program Counter), 스택 포인터 등 현재 상태를 PCB-A에 저장(Save)
  ↓
② 스케줄러가 다음에 실행할 프로세스 B를 선택
  ↓
③ PCB-B에서 프로세스 B의 이전 상태를 복원(Restore)
  ↓
현재: 프로세스 B 실행 중
```

**PCB(Process Control Block)**: 각 프로세스의 상태 정보를 담는 자료구조. 프로그램 카운터, 레지스터 값, 메모리 매핑, 열린 파일 목록, 프로세스 상태 등을 저장한다.

### 컨텍스트 스위칭의 비용

컨텍스트 스위칭은 공짜가 아니다. 상태를 저장하고 복원하는 과정에서 CPU 시간이 소모된다. 이 기간 동안 실제 유용한 작업은 전혀 실행되지 않는다. 이를 **오버헤드(Overhead)** 라고 한다.

스위칭이 너무 자주 발생하면 오버헤드가 누적되어 오히려 전체 성능이 저하된다. 운영체제 스케줄러의 핵심 역할 중 하나는 이 트레이드오프를 최적화하는 것이다.

스레드(Thread) 간 컨텍스트 스위칭은 프로세스 간 스위칭보다 훨씬 가볍다. 같은 프로세스의 스레드는 메모리 공간을 공유하므로 메모리 매핑 전환이 필요 없기 때문이다.

## 4. 시분할(Time Sharing) — 시간을 쪼개어 공평하게 나눈다

컨텍스트 스위칭이 *방법*이라면, 시분할은 *언제 전환할 것인가*를 결정하는 전략이다.

시분할은 CPU 시간을 작은 단위인 **타임 슬라이스(Time Slice)** 또는 **퀀텀(Quantum)** 으로 쪼개어 여러 프로세스에 순서대로 할당하는 방식이다.

```
시간 → ───────────────────────────────────────────────────▶
        [프로세스 A][프로세스 B][프로세스 C][프로세스 A][프로세스 B]...
        |← 퀀텀 →| |← 퀀텀 →| |← 퀀텀 →|
```

**라운드 로빈(Round Robin)**: 가장 기본적인 시분할 스케줄링 알고리즘. 준비 큐에 있는 프로세스들을 순서대로 동일한 시간 동안 실행한다. 퀀텀이 끝나면 현재 프로세스를 큐의 맨 뒤로 보내고 다음 프로세스를 실행한다.

### 퀀텀 크기의 트레이드오프

| 퀀텀 크기 | 장점 | 단점 |
|---------|------|------|
| 짧다 (수 ms) | 반응성 높음, 공평한 CPU 분배 | 컨텍스트 스위칭 오버헤드 증가 |
| 길다 (수십 ms) | 스위칭 오버헤드 감소 | 반응성 저하, 특정 프로세스 독점 위험 |

현대 운영체제는 보통 10~100ms 단위의 퀀텀을 사용하며, 우선순위(Priority)나 I/O 대기 여부에 따라 동적으로 조정한다.

## 5. 네 개념의 연결 구조

이 네 가지 개념은 독립적인 지식이 아니라 하나의 체계를 이룬다.

```
싱글코어 CPU
     │
     ▼
동시성(Concurrency) ──── 여러 작업을 동시처럼 처리하는 설계 목표
     │
     ├── 시분할(Time Sharing) ── 퀀텀 단위로 CPU 시간을 분배하는 정책
     │
     └── 컨텍스트 스위칭(Context Switching) ── 작업 전환 시 상태를 저장/복원하는 메커니즘

멀티코어 CPU
     │
     ▼
병렬성(Parallelism) ──── 실제로 동시에 여러 작업을 실행
     + 각 코어 위에 동시성도 함께 적용
```

- **동시성**은 목표다. "여러 작업을 효율적으로 처리한다."
- **시분할**은 정책이다. "CPU 시간을 어떻게 나눌 것인가."
- **컨텍스트 스위칭**은 메커니즘이다. "전환 시 상태를 어떻게 보존할 것인가."
- **병렬성**은 하드웨어 조건이 갖춰졌을 때의 가속이다.

## 마무리

사람이 싱글코어 CPU라는 비유는 정확하다. 우리가 여러 일을 동시에 잘 처리하는 것처럼 느껴질 때, 실제로는 빠른 전환과 집중의 교대가 일어난다. 그리고 그 전환에는 비용이 따른다. 이전에 무엇을 하고 있었는지를 기억하고, 다시 그 맥락으로 돌아오는 데 에너지가 소모된다.

CPU도 마찬가지다. 운영체제는 이 전환 비용을 최소화하면서도 모든 프로세스가 공평하게 CPU를 사용할 수 있도록 정교하게 설계된 스케줄러를 통해 동시성을 구현한다.

멀티코어 시대에도 동시성은 여전히 중요한 개념이다. 코어 수보다 실행 중인 스레드 수가 훨씬 많은 것이 일반적이기 때문이다. 병렬성이 처리량을 높이고, 동시성이 반응성을 유지한다. 둘은 경쟁 관계가 아니라 상호보완 관계다.
